## AI Safety Rules

These rules apply to every AI integration. A coding agent may not implement AI features that violate them.

1. **AI produces structured output only.** Every AI call returns a typed schema (Zod-validated). AI never produces raw SQL, raw HTML, or free-form instructions that the application executes directly.

2. **AI never generates or selects organization IDs, user IDs, or primary keys.** These are always generated by the database or resolved by the application from the authenticated session.

3. **Authentication and authorization are always deterministic code.** Never ask an AI model whether a user is allowed to perform an action. That check is always in TypeScript.

4. **Destructive and sensitive changes require explicit human confirmation:**
   - Allergen declarations → `allergens_confirmed = false` until the owner clicks Confirm
   - Price changes → require re-save (not auto-applied from AI suggestion)
   - Publishing a menu → explicit Publish button, not automatic
   - Bulk import → Preview step required before commit; 24-hour rollback available

5. **Service-role Supabase client is behind a named domain service.** It is never instantiated in a React component, a generic utility, or an API route that handles untrusted input. Only files in `lib/db/service-role.ts` use it, with one function per operation.

6. **PII boundary:** Menu content (item names, descriptions, ingredient lists, prices) is business data — acceptable to send to AI APIs. The following are PII and must never be sent to any external AI API: customer names, customer phone numbers, customer order history, owner's personal phone number if different from business number, WhatsApp message content that names customers.

7. **AI audit trail is mandatory.** Every AI call writes to `ai_audit_events` before the result is applied. Fields: provider, model, prompt_version, SHA-256 input hash (not raw input), output_summary, approved_by, approved_at.

8. **AI-generated food images require approval and provenance.** Image source stored as `image_source = 'ai_generated'` in `menu_items`. Restaurant owner must explicitly approve before image appears on public menu.

9. **Prompt versions are tracked.** Each distinct system prompt has a semantic version slug (e.g., `allergen-v1.0`). When a prompt is changed, the version is bumped and recorded in `ai_audit_events`.

## Authorization Pattern

RLS enforces data isolation between tenants. Application code enforces what authenticated users of the same tenant can do.

```typescript
// Pattern for every Server Action that mutates data:

export async function updateMenuItemPrice(itemId: string, priceHalalas: number) {
  // 1. Authenticate
  const session = await getRequiredSession() // throws if unauthenticated

  // 2. Resolve org membership and role
  const membership = await db.query.organizationMemberships.findFirst({
    where: and(
      eq(organizationMemberships.userId, session.user.id),
      inArray(organizationMemberships.role, ['owner', 'manager'])
    )
  })
  if (!membership) throw new AuthorizationError('Insufficient role')

  // 3. Verify item belongs to this org (defense in depth — RLS also enforces this)
  const item = await db.query.menuItems.findFirst({
    where: and(
      eq(menuItems.id, itemId),
      eq(menuItems.organizationId, membership.organizationId)
    )
  })
  if (!item) throw new NotFoundError()

  // 4. Validate input
  const validated = priceSchema.parse(priceHalalas) // Zod

  // 5. Mutate
  const [updated] = await db
    .update(menuItems)
    .set({ priceHalalas: validated, updatedAt: new Date() })
    .where(eq(menuItems.id, itemId))
    .returning()

  // 6. Audit
  await writeAuditEvent({
    organizationId: membership.organizationId,
    userId: session.user.id,
    action: 'menu_item.price_changed',
    entityType: 'menu_item',
    entityId: itemId,
    beforeSnapshot: { priceHalalas: item.priceHalalas },
    afterSnapshot:  { priceHalalas: updated.priceHalalas }
  })
}
```

## NCA — National Cybersecurity Authority

*Source: NCA ECC-1:2018 and NCNICC-1:2025 public documents. Status: assumption based on public text — requires NCA-licensed consultant review before government or financial-sector customers are onboarded.*

| Control Domain | V1 Implementation |
|---|---|
| Identity & Access | MFA available via Supabase Auth (TOTP); enforce for all admin accounts |
| Data Encryption | TLS 1.3 in transit (Cloudflare); AES-256 at rest (Supabase managed) |
| Logging | `audit_events` table (immutable); GitHub Actions run history; Supabase auth logs; retain ≥ 12 months |
| Secret Management | GitHub Secrets + Cloudflare Worker secrets; `::add-mask::` in CI; TruffleHog on every PR |
| Incident Response | Written plan required before launch; 72-hour notification window |
| Vulnerability Management | Semgrep SAST on every PR; Dependabot (daily); npm audit in CI |
| Backups | Daily (Supabase Cloud managed); monthly restore test |

**Certification path:** ISO 27001 at ~SAR 400K ARR (unlocks enterprise and government chain customers).

## SFDA — Digital Menu Mandate

*Source: SFDA Circular on Digital Menu Labeling, effective July 1, 2025. Status: verified by public SFDA publication.*

| Requirement | Implementation |
|---|---|
| Calories per serving | `calories INT` field in `menu_items`; displayed on public menu |
| Sodium content | `sodium_mg INT` field; displayed on public menu |
| Caffeine content (beverages) | `caffeine_mg INT` field; displayed on public menu |
| Allergen declarations | `allergens TEXT[]` field; `allergens_confirmed BOOLEAN`; AI-suggested, human-confirmed |

**AI and SFDA:** Groq allergen detection suggests allergens from ingredient text. The suggestion is never auto-published. It is shown to the owner with a Confirm button. The `allergens_confirmed = false` flag prevents unconfirmed allergens from displaying on the public menu.

## CI/CD Security Controls (V1)

The V1 pipeline is intentionally minimal. Advanced scanning is a later maturity phase.

### Every PR (`ci.yml`)
- ESLint + Prettier check
- TypeScript strict mode (`tsc --noEmit`)
- Vitest unit tests
- Drizzle migration lint (no destructive changes without `-- BREAKING` annotation)
- TruffleHog secret scan (since HEAD~1)
- Gitleaks pre-commit hook (local, not CI-only)
- Production build check
- Lighthouse CI (performance ≥ 80, accessibility ≥ 90)
- Semgrep SAST (auto config — covers Next.js, TypeScript, SQL injection, XSS)

### Every merge to main (`deploy.yml`)
- Integration tests (Supabase local Docker): auth, RLS tenant isolation, import_jobs rollback
- Deploy to staging
- Smoke test (health endpoint returns 200; public menu page loads)
- Deploy to production (10-minute wait timer on GitHub Environment)

### Monthly (manual)
- Database restore test from backup
- Review and rotate any secrets approaching 90 days

### Later maturity phase (not V1)
- OWASP ZAP full active scan
- Nuclei CVE templates
- Container image scanning (Trivy) — add when self-hosted services are introduced
- Nightly security automation

**Statement:** Automated scanners detect categories of known vulnerabilities. They do not replace penetration testing or human code review. Schedule a penetration test before handling sensitive customer financial data (V3).
